Flex Work 2.0 · Architecture · v1.6

Flex Work 2.0 Plan

Scope. Three sequential phases delivered from a single code path: a refreshed frontline VMS, an embedded Dayforce workforce module, and expansion to Assignment, Project, and SOW engagement types. Architecture. A type-blind data model; one frontend serving buyer, agency, and worker; four engagement types; every change gated by feature flags so existing tenants are unaffected until an admin opts in. Independent infrastructure, integrated surface. Flex Work runs on its own Rails monolith, Postgres database, and deploy pipeline. It adopts Dayforce data models for every shared primitive and embeds inside Dayforce Core via an iframe wired with SSO, theme handoff, and postMessage events. Customers experience a single product; the two systems release on independent cadences.

01 · The three-phase strategy

Three sequential phases. One code base supports all of them.

Each rung is a distinct go-to-market with a larger addressable market than the last. v2 must support all three from a single code base. An architecture that fits only the first rung forces a re-platform to reach the second — typically an 18-month setback. The work is sequenced so each phase ships incremental value while the foundations for the next phase are already in place.

Rung 1 · Current

Deliver frontline VMS.

High-volume agency shift requisitions, multi-supplier distribution, site-manager timesheet approval. This is the core workload of today's Flex Work, rebuilt on the v2 foundations. Frontline is the established market position; Rung 1 protects and extends it.

Architectural debt this clears: ad-hoc LOCATIONS / WORKERS tables, free-text job titles, supplier sidecars duct-taped to requisitions, ledger codes nobody can read.
Rung 2 · Next

Embed as a Dayforce workforce module.

A Flex Work entry in Dayforce Core's left nav opens the Flex Work app inside an iframe. SSO is handled, the chrome is themed, and postMessage bridges cross-frame events. Contingent spend appears on the same GL Preview as badged spend, site managers approve from a single queue, and credentials and Pay Codes flow in both directions. Customers see one product; the two code bases remain independent.

What this needs: Flex Work tables must adopt the Dayforce data models for OrgUnit, Job, JobAssignment, Position, WorkAssignment, TimePair, LaborMetricCode — kept in sync by API connectors, never forked. Plus a hardened iframe contract (SSO, theming, deep-linking, postMessage events) so the embed feels native.
Rung 3 · Expansion

Expand to the broader VMS market.

Beyond frontline shifts: Professional staffing, Statement-of-Work agreements, Project-burn delivery, Independent Contractors, Employer of Record. Enterprise buyers procure a vendor management system that covers every engagement type, not a frontline-only tool.

What this needs: a type-blind data model — one Requisitions list, one Workforce list, one Timesheets list, one Invoices list — that supports every engagement type without forking the product into per-type pages.
Architectural premise

One foundation, three go-to-market motions, two cleanly separated code bases. There is no separate "Flex Work Pro" or "Flex Work SOW" product line; engagement types are universal scopes on the existing surfaces, gated by feature flags. Flex Work does not merge into Dayforce Core; it adopts Dayforce data models and embeds via iframe, releasing on its own cadence. A customer that buys frontline today and SOW in 2027 does not migrate — they enable a flag.

02 · Architectural decisions

Six load-bearing decisions. Every later section follows from these.

Each card states the decision, its consequence for the product, and the contract it imposes on the implementation. The two columns serve different readers without splitting the document: Product for stakeholders evaluating impact, Implementation for engineers writing the code.

PRINCIPLE 01

A type-blind data model. Universal scopes, not per-type pages.

One Requisitions list. One Workforce list. One Timesheets list. One Invoices list. Engagement Type (Shift, Assignment, Project, SOW) is a filter on these surfaces, never a separate destination. The buyer learns the application once.

Product
Enabling a new engagement type for an existing tenant is a feature-flag change, not an onboarding flow.
Implementation
Every list page accepts an engagementType axis. No new top-level routes per type. Engagement Type resolves from sourcingChannel plus an explicit field; it is never hard-coded per page.
PRINCIPLE 02

Adopt Dayforce data models. Keep the code base independent.

Flex Work runs on its own Rails monolith, its own Postgres database, and its own deploy pipeline. Every table that has a Dayforce equivalent adopts that data model — same shape, same field names, same enum values — and stays in sync via API connectors. Where Dayforce has no equivalent primitive (Supplier, SupplierContract, BillingLine, Invoice), Flex Work owns a top-level table that references the Dayforce-shaped row by id and never forks from it.

Product
Finance reports show contingent spend on the same GL Preview as badged spend. Customers receive a Dayforce-native experience while Flex Work releases on its own cadence.
Implementation
Sibling pattern: ContingentEngagement keyed by positionId, BillingLine keyed by timePairId, SupplierInvoiceLine keyed by billingLineId. Ids on mirrored tables are Dayforce-issued and replicated; Flex Work does not mint its own id space for any mirrored row.
PRINCIPLE 03

Feature flags are the migration mechanism.

Every behavioural change ships behind a flag. Existing tenants remain functionally identical to the prior release until an admin opts in. Flags are scoped per Business Unit for structural migrations (org tree, position management) and per Org for engagement-type axes.

Product
Existing tenants receive zero-disruption upgrades; opt-in is the customer's decision, not a forced migration.
Implementation
getFeatureFlag(key) reads from flexwork.featureFlags; engagement-type axes live at flexwork.engagementTypes.{orgId}. Every flag has a default-off contract: with the flag off, behaviour must be functionally identical to the prior release.
PRINCIPLE 04

Reference Codes on every object, from day one.

Every top-level object carries a Reference Code (unique, no special characters such as hyphens). Every Labor Metric Code carries both a Reference Code and a Ledger Code. Without these, rows cannot round-trip through standard import/export pipelines and cannot appear on GL Preview or Earning Statements.

Product
Implementation timelines measured in weeks rather than months. Existing customer imports work without bespoke transforms.
Implementation
Validate at write time. Distinguish user-defined Reference Codes from System-Defined codes (EEO-1 Job Category, FLSA Status, KldB Job Classification). Surface the code in every list-page row.
PRINCIPLE 05

One frontend. Multiple roles, multiple tenants.

A single React application serves buyer, agency, and MSP-program tenants. A viewAs role dimension swaps navigation, omnibar scope, and permitted actions. Cross-tenant identity is bridged by a SupplierTenantLink record over Dayforce SSO, not by a separate product.

Product
Staffing-agency tenants (for example, StaffWise) are an extension of the existing product, not a separate build. MSP programs are configuration, not code.
Implementation
navItemsForRole(role) is the single switch for top-nav variance. Role and Role Features are Dayforce-native; supplier-side experiences are scopes of the same User Role table.
PRINCIPLE 06

Embed in Dayforce Core as an iframe. Integrated surface, separate engine.

Inside Dayforce Core, a Flex Work entry in the left nav opens the Flex Work application in an iframe. SSO is handed off via a signed Dayforce token; Everest tokens propagate into the iframe so the chrome matches the host theme; deep-link routes are wired in both directions via postMessage; the host-page chrome is suppressed inside the iframe so the boundary is not visible. Customers experience a single product; the two code bases release on independent cadences.

Product
Customers receive a unified Dayforce experience. Flex Work releases on its own schedule without coordinating with Dayforce Core release trains.
Implementation
Frame contract: parent.postMessage({type:'fw:navigate',path}) for outbound deep links; iframe.contentWindow.postMessage({type:'fw:sso',token}) for the handshake. X-Frame-Options: ALLOW-FROM dayforce.com plus frame-ancestors CSP locked to Dayforce origins. Detect ?embed=dayforce and suppress the top nav and footer; render the dense embed shell in their place.

Why the engagement-type matrix precedes the data model

The four engagement types are the complete demand surface v2 must support. Every architectural decision in §02 and §05 is evaluated against the test: does it support all four cleanly, or does it bias the system toward Shift-only? Decisions that bias the system are excluded.

05 · Core object model

Three buckets, all in Flex Work's database: net-new, extended, replicated.

Every table below lives in Flex Work's own Postgres — nothing rides in Dayforce Core. What varies is the data model and who is the system of record. Five net-new tables carry supplier economics that Dayforce has no concept of. Five extended tables adopt a Dayforce schema and add VMS-specific columns. Everything else is a thin replica of a Dayforce table, kept in sync by API connectors. The next section names the source-of-truth Dayforce path per module.

Net-new · Flex Work owns the model

Supplier economics. Owned by Flex Work end-to-end.

Dayforce has no equivalent for any of these objects. They live exclusively in Flex Work's database; foreign keys reference the Dayforce-shaped rows that Flex Work already replicates.

  • SupplierTop-level third-party org · modeled on Plan Carrier · linked to Legal Entity
  • SupplierContractRate cards, markups, tenure rules · output drives bill rate
  • SupplierInvoice + SupplierInvoiceLineGenerated from approved time pairs · references orgUnit / jobAssignment / laborMetric / employee
  • DistributionRule + SupplierTierTiered routing to the supplier panel
  • PricingRule + MarkupRuleRule engine — output writes to ContingentEngagement.billRate
  • SOWAgreement + MilestoneAcceptanceSOW engagement type
  • ProjectBudget + BurnLedgerEntryProject engagement type
Extended · adopts Dayforce data model + adds VMS fields

Same shape as Dayforce, with VMS-specific extensions.

Flex Work owns the row, but the schema mirrors the Dayforce primitive one-for-one and adds a small set of contingent-specific columns. The shared columns sync via API connectors; the extension columns remain private to Flex Work.

  • ContingentEngagementKeyed by positionId · supplier, billRate, markup, contractId, poId
  • BillingLineKeyed by timePairId · billRate, markup, billAmount, invoiceId
  • FlexWorkLocationProfileSidecar by orgUnitId · Invited lifecycle, supplier-side aliases
  • SupplierTenantLinkCross-tenant agency identity bridge
  • Employee.contingentEngagement sub-recordActive engagement only — set on ContingentWorker type
  • Position.distributionListSupplier routing list per Position
  • JobRequisition.flexWorkOrderIdMulti-role Flex Work order grouping
Replicated · Dayforce is system of record

Flex Work holds a synchronized copy.

Authority lives in the Dayforce module's screen. Flex Work maintains an API-synced replica to support local joins, rendering, and routing without a round-trip. Writes inside Flex Work queue an outbound call to the Dayforce primitive; inbound webhooks keep the replica current. See §07 for the full mapping.

  • OrgUnit (Site)Org Setup > Organization
  • Job + JobAssignmentOrg Setup > Jobs and Job Assignments
  • Position + WorkAssignmentOrganization Designer · People > Work
  • TimePair + PayCodePay Setup > Time Entry Policy
  • LaborMetricCodeOrg Setup > Labor Metrics
  • JobRequisitionRecruiting · Job Requisitions
  • Employee (+ workerType, sourcingChannel extension)People
Implementation heuristic

If a row needs to appear on a Pay Statement, in GL Preview, in HR Headcount, or in Recruiting, it must adopt the Dayforce data model (replicated or extended) so the API connector can write it back cleanly. The Net-new bucket is reserved for objects that handle money flowing out of the buyer to a third-party supplier — a flow Dayforce Payroll does not generate.

The shape of every integration

Every card above follows the same physics: Authenticate once, then Query for reads and Execute for writes — all driven by typed Request classes, all returning typed Responses. Real-time deltas push through the Receive endpoint and recover via the pull-side notification API. XRefCode is the universal join key; NotificationDataId threads each event back to the data it touched. If a future integration doesn't reduce to those three verbs and those two identifiers, it's not actually a Dayforce integration — it's a one-off the connector team will refuse to ship.

The two-system contract, summarised

The iframe is the integration surface; the connector is the data wire. The frame contract is strict: if any of the listed postMessage events fail, the boundary between the two systems becomes visible to the user. The connector contract is resilient: queued, idempotent, and retried. A Dayforce Core outage must not bring down Flex Work's agency-facing experience, and a Flex Work deploy must not interrupt Dayforce Payroll. Two independent code bases, two independent SLAs, one product to the customer.

12 · Tech stack

Concrete versions, pinned. One language family from API to mobile.

The stack is selected so a single engineering team can deliver buyer web, agency web, and a worker mobile application without partitioning into three independent codebases. Ruby on Rails on the backend, React on the web, React Native (via Expo) on mobile, and a shared TypeScript type layer between them. Every version listed below is the minimum supported — production runs at or ahead of this floor, and the team upgrades forward as LTS bumps land. The stack is deliberately leading-edge: Rails 8 with the Solid Trifecta (Solid Queue, Solid Cache, Solid Cable on Postgres), Ruby 3.4 with the Prism parser and YJIT, React 19 with Actions and async transitions, React Native 0.76 with the New Architecture (Fabric + TurboModules + Hermes) on by default.

What "monolith" means here, precisely

Monolith refers to the deployment unit and the code organisation, not the runtime topology inside a process. The API and the web SPA ship as one Docker image. The shared TypeScript surface keeps web and mobile in lock-step. The Rails process itself runs multiple components in parallel — Puma for HTTP, Solid Queue for jobs, Solid Cable for WebSockets — but they boot from one codebase, share one database, and deploy together. Mobile is in the monorepo for shared code and CI, but distributed separately because Apple and Google require it.

15 · AI-optimized build order

Twelve steps. Sequenced so each step constrains the next.

AI-assisted development is most effective when the specifications it operates against are precise, machine-readable, and already in place. The sequence below maximizes the surface that a model can generate before any production UI is written by hand: every step inherits its constraints from earlier steps, and never the reverse. Tasks within a step may run in parallel; steps between are ordered.

Step 01

Design system

Everest tokens (colors, type, spacing, radius, elevation), Clarika fonts, the Flex Work UI kit on top of Everest. The artifact is tokens.css + the everest/ primitives + a Storybook of every component variant.

Why first. Every downstream screen pulls from this layer. No invented hex values, no off-grid spacing, no parallel component libraries.

  • tokens.css + dark-mode mirror
  • Primitive component library (Button, Field, Table, Modal, Toast)
  • Everest icon set + logo
  • Storybook with usage rules per component
Step 02

API docs & OpenAPI spec

The full REST + GraphQL contract for v2, defined before a single endpoint is implemented. Lives at Flex Work API Reference.html; the canonical machine-readable form is the OpenAPI 3.1 spec in api/openapi.yaml.

Why second. Backend controllers, TypeScript types, MSW mocks, and the Postman collection all generate from this single artifact. Drift between any of them constitutes a defect.

  • OpenAPI 3.1 spec (canonical)
  • HTML reference (this doc)
  • JSON Schema for every request/response
  • Generated MSW handlers + Postman collection
Step 03

Database schema & migrations

Every table from §19 (Database schema) as a Rails migration. Indexes, FK constraints, partitioning declared upfront. db/structure.sql committed and reviewable.

Why third. Models, factories, and seed data all depend on a stable schema. Roughly 80% of the migration boilerplate can be generated directly from the schema diagram in §19.

  • Rails migrations for every Mirror / Extend / Net-new table
  • Partitioning DDL for hot tables (TimePair, BillingLine)
  • db/structure.sql
  • Seed data for a "Hospitality Group" reference tenant
Step 04

ActiveRecord models & domain logic

One model per table, with validations, scopes, and associations matching the API spec exactly. Service objects for the cross-table logic (pricing, distribution, invoice generation).

Why fourth. The schema and API specification together fully constrain model code. Models, validations, and FactoryBot definitions can be generated in a single pass against those two specs.

  • app/models/ for every table
  • app/services/ for pricing, distribution, invoicing
  • FactoryBot factories
  • RSpec model + service specs
Step 05

API endpoints (controllers + GraphQL resolvers)

Implement every endpoint in the OpenAPI spec. Contract tests assert request/response shapes match the spec byte-for-byte. Authorization via Pundit policies, one per resource.

Why fifth. With the specification and models in place, controllers are integration glue. Contract tests pinned to the OpenAPI spec detect any drift between specification and implementation automatically.

  • app/controllers/api/v2/ for REST
  • app/graphql/ resolvers
  • Pundit policies per resource
  • Contract tests pinned to api/openapi.yaml
Step 06

Background jobs & Dayforce connectors

Sidekiq jobs for the queued, idempotent work: distribution cascades, invoice generation, GL Preview push, and the Dayforce API connectors named in §08. Inbound webhooks land here too.

Why sixth. Models and services have stabilized; background jobs orchestrate them. Each connector is small, isolated, and well-suited to scaffolded generation against the relevant Dayforce Web Service contract.

  • Sidekiq workers per cross-table operation
  • Dayforce API connector classes (one per replicated table)
  • Webhook handlers
  • Reconciliation jobs (nightly drift check)
Step 07

Shared TS types & SDK

Generate shared/api-types.ts from the OpenAPI spec; hand-write a thin shared/sdk/ wrapping TanStack Query hooks. Both web and mobile consume this — no parallel client code.

Why seventh. Provides a single typed surface for every client. Schema changes propagate through regeneration rather than coordinated search-and-replace.

  • shared/api-types.ts (generated)
  • shared/sdk/ (TanStack Query hooks)
  • shared/zod/ (form schemas mirroring API)
  • MSW mocks for offline dev
Step 08

Prototype-to-production parity pass

Walk every prototype screen in index.html; for each, produce the production React component using the design system primitives (step 01) and the SDK (step 07). The prototype is the spec; the production component is the implementation.

Why eighth. The interaction design is already resolved in the prototype. Translating a working prototype into production code against a fixed component library and SDK is a well-bounded task — but only once both endpoints exist.

  • Production React components for every prototype page
  • Storybook stories matching prototype states
  • Playwright tests asserting prototype-equivalent behavior
Step 09

Mobile (worker app first)

React Native worker app reusing the SDK from step 07 and the design tokens from step 01. Clock-in/out, schedule, credentials, pay — the canonical worker loop.

Why ninth. Mobile reuses approximately 70% of the types, hooks, and tokens established earlier. Building mobile after the web client means the shared SDK has already been exercised in production.

  • Expo Router screens for the worker loop
  • Offline cache via MMKV
  • EAS Build + Update pipeline live
Step 10

Dayforce iframe embed

The host-side wiring described in Principle 06 — SSO handshake, theme handoff, postMessage deep-link bridge, the embed-mode chrome that hides nav/footer when ?embed=dayforce is set.

Why tenth. The embed is a thin integration shell over the production application. Building the application first and adding embed support second is materially simpler than the reverse.

  • SSO handshake endpoint + JWT exchange
  • postMessage event bus
  • Embed-mode CSP + frame-ancestors config
  • Theme bridge (Everest tokens to iframe)
Step 11

Feature-flag & rollout scaffolding

LaunchDarkly per-org/per-BU flag plumbing, the dev-mode fallback, the admin UI that lets a customer's admin opt into a flag. Every flag has a default-off contract and a sunset date.

Why eleventh. Flags wrap completed features, not future ones. Adding flags immediately before rollout, rather than as preemptive scaffolding, keeps the flag inventory tractable.

  • LaunchDarkly server-side eval middleware
  • Per-org / per-BU flag store
  • Admin UI for flag toggling
  • Flag-off byte-equivalence audit
Step 12

E2E + visual regression + load

Playwright flows that cover the §09 lifecycle end-to-end on a seeded reference tenant. Chromatic for visual regression. k6 load tests for the hot paths (distribution, invoice generation, timesheet approval).

Why last. Every preceding step produces what tests verify. Test scaffolding inherits its shape from the implementation it covers.

  • Playwright E2E for each lifecycle stage
  • Chromatic baseline for every Storybook story
  • k6 scripts hitting /timesheets, /invoices, /distributions
  • CI gate: regression = block merge
Sequencing rule within a step

Within any step, tasks are ordered by how much downstream code they constrain. The most-constraining artifact comes first; leaves come last. Within step 04, models precede service objects; within step 05, REST controllers precede GraphQL resolvers (REST is the more constrained generation target). The principle matches the step ordering at large: establish constraints before generating code against them.
