What this is. A role-by-role inventory of the work needed to bring Flex Work's worker on-/offboarding flow up to the bar set by SAP Fieldglass and Workday VNDLY for Professional engagements, and Joined Up for Frontline direct-sourced shift work. The Contractor (IC) lifecycle shipped on its own parity scope and is excluded.

Flex Work VMS · On-/Offboarding parity · v0.2 · implementation status

Bringing the worker lifecycle to Fieldglass, VNDLY, and Joined Up parity — without forking a parallel app.

Flex Work today onboards three of its four engagement types with intent: Contractors run a six-step direct wizard, Professionals run a four-step buyer-invited panel, and SOW resources arrive supplier-managed. What's missing — and what this list closes — is the Frontline lifecycle (Joined Up parity: direct-invite, I-9/RTW, W-4, orientation, shift-ready), the Professional offboarding story end-to-end (Fieldglass + VNDLY parity: end-engagement reason, asset return, access revocation, exit interview, knowledge transfer), and the shared infrastructure both engagement types lean on (worker lifecycle status, audit, webhooks, approval router). The data layer and detail surface for both engagement types are already in place inside pages/workforce.jsx and pages/worker-mobile.jsx — the work below extends them in-place rather than adding a parallel "onboarding" page or model.

01 · Scope & methodology

Two engagement types, three benchmarks, one shared backbone.

For this list, onboarding is the path from worker is selected to worker can earn — every form signed, document collected, check cleared, access provisioned, orientation completed. Offboarding is the path from engagement ends (voluntary or involuntary) to access revoked — last shift handled, assets returned, final pay reconciled, do-not-rehire flag set when warranted. Tasks split by engagement type because the bar is set by different products: Professional onboarding maps to Fieldglass + VNDLY's invite→tax→BG→NDA→asset→orientation flow; Frontline maps to Joined Up's W‑2 direct-sourcing shape — invite code, I-9/RTW + W-4, orientation video, shift-ready in days not weeks.

Out of scope

Contractor (IC) lifecycle shipped end-to-end on Contractors parity v2.0 — classification gate, MSA + tax-form + banking onboarding, convert-to-employee handoff, year-end 1099 e-file. SOW resources are onboarded by the supplier under the SOW agreement; the buyer's lifecycle bookend is the SOW closeout, not a per-worker exit. Agency-supplied Frontline workers are onboarded by their agency — Flex Work's job there is the compliance gate (credentialing, I-9 sighted, banking on supplier side), not running the workflow.

What counts as parity

A task is parity-bound if a Fieldglass, VNDLY, or Joined Up customer can run the equivalent workflow today and we cannot. Tasks tagged Beyond are intentional extensions — native Dayforce HR handoff, real-time co-employment status, AIX assists on document review — that none of the three reference platforms own end-to-end. The bar is functional parity, not visual mimicry.

Architectural discipline

Per Architecture Principle 01, on-/offboarding is a filter on every Flex Work surface, not a sibling product. Every task in this list maps to an existing page or model — section 03 names the host. The work extends pages/professional-onboard.jsx (already hosts AddProfessionalPanel + RenewProfessionalPanel), pages/remove-worker-panel.jsx (already does shift-scoped removal, scopes up to engagement / worker), pages/worker-mobile.jsx (gains an Onboarding tab pool-conditionally), pages/workforce.jsx (status enum + accordion), and pages/settings-config.jsx (Pro + Frontline program cards alongside the existing SOW and Contractor cards). Net new top-level files: zero.

02 · Where we stand

Both bookends now run end-to-end on the happy path. Polish, routers, and shared-infra events are the remaining work.

At the start of this scope, Professional onboarding shipped only as the front of the AddProfessionalPanel flow, Frontline had no buyer-initiated direct path, and offboarding didn't exist beyond shift-scoped removal. That has changed. Professional and direct-sourced Frontline now share a configurable lifecycle: the Add panel carries a Frontline variant, both engagement types read a per-role / per-country task catalog from the two program cards in Settings, the worker detail shows a live onboarding/offboarding tracker, the worker-mobile shell surfaces an On-/Off-board tab, and offboarding runs through the scoped RemoveWorkerPanel (now shift | engagement | worker) with reason, effective date, last-shift handling, asset-return, and final-pay. The worker lifecycle status enum gained Offboarding, Inactive, and Terminated. What's left is the polish and plumbing tier: the two approval-router presets, the schedule.jsx last-shift booking gate, the three lifecycle webhook transition events, the document library, the offer-accept auto-trigger, agency→direct conversion, and bulk offboard — none of which block the core flow.

03 · Architecture · embed, don't add

Every task lives in an existing page or model. Net new top-level pages: zero — one shared config module added.

On-/offboarding is a filter on every Flex Work surface, not a sibling product. The work extends Settings, Workforce, Worker mobile, the remove-worker panel, and the professional onboarding panel — rather than fork a parallel "Lifecycle" app. The mapping below names the existing host file for every task and the existing primitives it reuses. Any task that would introduce a new top-level file is flagged so we can push back early.

How it actually landed. Five existing files carried the work as planned: pages/professional-onboard.jsx (the Add panel gained a workerType: "frontline" | "professional" variant flag branching step 3), pages/remove-worker-panel.jsx (scope lifted from shift-only to shift | engagement | worker, with new lifecycle-mode steps RwStepLifecycle / RwStepLifecycleClose / RwStepLifecycleReview), pages/worker-mobile.jsx (a pool- and status-conditional Onboarding/Offboarding tab via WmOnboardingScreen), pages/workforce.jsx (the WF_STATUS_HUES status enum plus the LifecycleTrackerBody accordion), and pages/settings-config.jsx (renders the two program cards). The one deviation from the original “zero new files” plan: the task catalogs, connector registry, and reader helpers (getOnboardingTasks / getOffboardingTasks / getLifecycleConnectors) were factored into a single new data + config module, pages/lifecycle-config.jsx, rather than duplicated across the five consumers. It registers no route and renders no standalone screen — it exposes the two configuration cards and the shared catalog the other surfaces read. This kept the program definition single-sourced; the alternative (inlining the catalog into settings-config.jsx) would have forced every consumer to reach into the Settings page for data. Net new top-level routes / pages: zero.

04 · Admin · configure the lifecycle

Admin authors the onboarding and offboarding programs once; every downstream surface reads from them.

An admin's job is to set the contract that on-/offboarding runs under — what tasks ship by default, who has to approve, which connectors handle the I-9 / BG-check / e-Sig call, which documents go out for signature. Every Manager-side and Worker-side surface in section 05 / 06 reads from the Admin config — nothing is hand-rolled per requisition. The work lands in the existing pages/settings-config.jsx Configuration hub, slotted next to the SOW + Contractor program cards already there.
