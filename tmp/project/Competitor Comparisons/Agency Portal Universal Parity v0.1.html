01 · Scope & methodology

The agency is a supplier. Both worlds run the same loop.

This list covers the Agency persona only — the StaffWise tenant (kind: "agency"), viewing as Agency, scoped to its agencySupplierId. The benchmark is functional parity, not visual mimicry: we match what an agency can do in a competitor today, expressed in the Everest language and the Flex Work data model. Two competitor worlds set the bar.

The two benchmark worlds

Frontline / shift — JoinedUp by Beeline. The agency's job is to fill volume fast: broadcast an open shift to a qualified bench, take the first accept, clear compliance, get the worker on site. The portal is availability-, coverage-, and SLA-led.
Professional — Vndly + Fieldglass. The agency's job is to win the role: see the open requisition, submit a ranked shortlist with a proposed rate, move candidates through interviews, get scored on quality and responsiveness. The portal is submission-, status-, and scorecard-led.

The architectural discipline

Per Architecture Principle 01, engagement type is a filter on every surface, not a sibling product. This list holds the agency portal to the same rule: no per-engagement-type pages, no per-engagement-type nav. Where frontline and professional need different behaviour, that difference is a branch inside a shared surface — the same way assign-worker-panel.jsx already swaps Assign ↔ Submit by engagement type. Every task below names its host file. Zero new top-level routes; zero new nav entries.

03 · The universal agency model

Receive demand · respond with supply · get measured · get paid.

Strip the vocabulary away and JoinedUp and Vndly run the agency through the same four steps. Only the Respond step truly differs by engagement type — and that difference is already built, as the allocate / submit / propose modes that AssignmentEngine.modeFor(engType) selects from the requisition's type. We don't design three flows; we surface the one spine that already exists, and let the ordered job pick the variant.

Step 1

Receive demand

An open requisition lands with the agency — distributed by tier, or direct.

Shift / block to fill by a window Role to submit candidates against Team shape to propose against a budget
Step 2

Respond with supply

The one place the worlds diverge — and the one we already model as a mode.

Assign / broadcast from bench Submit ranked shortlist + rate Propose a team + plan
Step 3

Get measured

The buyer scores the agency; the agency needs to see the same numbers.

Fill rate · no-show · attendance Submit→hire · responsiveness · quality On-budget · on-time · acceptance
Step 4

Get paid

Time and expense roll into a consolidated, exception-clean invoice.

Weekly consolidated timesheet Period / milestone invoice Burn-based progress billing
S Shift / frontline P Professional J Project
The spine already exists — and it is job-driven, not hardcoded

This is the most important fact in the document, and it is a constraint on every task below. The Respond step is not branched on “frontline vs professional” anywhere in the codebase. assign-worker-panel.jsx reads one value — AssignmentEngine.modeFor(cfg.engagementType) — which maps Shift → allocate, Assignment → submit, Project / SOW → propose. The same engine already owns the credential pre-check (credentialSummary), the availability guard (availabilityFor), the onboarding lifecycle (onboardingItemsFor), bulk allocation, the rate band, and the offer/accept loop (createOffer / respondOffer). Role only flips labels (agency “Assign” vs buyer “Request”) — never behaviour. Every recommendation in §04 must route through this engine and read the engagement type from the requisition; no task may add a frontline or professional code path of its own.

The design rule that falls out of this

Each of the four steps becomes a single surface in the agency workbench, and inside it the engagement type chooses the variant. The Inbox shows fills and submissions and proposals in one ranked list. The Respond action is one button with three bodies. The scorecard shows one composite plus the per-world metrics that matter. The agency never learns three products — it learns one, and the work it does decides what it sees.

Net architectural footprint of the build

Zero new top-level files. Zero new pages. Zero new nav entries. Zero new routes. Zero schema additions. The hardest part — the job-driven Respond spine — already existed in assignment-engine.jsx, so nothing added a frontline or professional code path. The new agency surfaces mounted into dashboard.jsx, workforce.jsx, timesheets.jsx, invoices.jsx, assign-worker-panel.jsx and agency-pro.jsx behind an agency-tenant check; the styles rode in the existing styles-agency-pro.css. Every surface reuses entities already in the model: Worker, Requisition, SupplierContract, SupplierTier, DistributionRule, Timesheet, and the credential store. New persistence is confined to a few localStorage keys (bench additions, per-worker availability, HCM-connected flag).
