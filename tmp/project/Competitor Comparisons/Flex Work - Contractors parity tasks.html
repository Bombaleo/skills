What this is. A role-by-role accounting of the Contractors (1099 / IC) module against the three reference platforms the field competes against — SAP Fieldglass IC, Workday VNDLY, and Deel. The original v1.0 plan listed eighteen open tasks across three releases; this v2.0 closes the log: every task is shipped, every file is named, and the architectural footprint stayed inside the "no new top-level pages" rule.

How to read it. Roles match the four canonical Flex Work personas adapted for the IC engagement type: Admin (Workforce Ops · Legal · Finance · IC Compliance), Manager (hiring manager · engagement owner), Contractor (the IC themselves — the worker variant for IC, distinct from agency-supplied Worker), and Shared (infra, webhooks, AIX). Every shipped task carries the file it lives in.

01 · Scope & methodology

Contractors means direct-engaged 1099 / IC — no agency intermediary, full classification + tax compliance burden on us.

For this list, a Flex Work Contractor is a worker with pool === "Contractor", no SupplierLink, an IndependentContractorEngagement (the MSA + SOW pair already shipped in pages/contractors.jsx), a tax form on file (W-9 / W-8BEN / W-8BEN-E), and a classification determination (IRS 20-factor + ABC, already shipped). Anything reached via a supplier or agency is out of scope — that workflow lives in Frontline and Professional. Statement-of-Work engagements (work-product–scoped, MSA-anchored, agency-delivered) are also separate; the contractor-SOW pair shipped here is the IC variant where the contractor is the SOW counterparty.

What counts as parity

A task is parity-bound if a Fieldglass IC, VNDLY, or Deel customer can run the equivalent workflow today and we cannot. Tasks tagged Beyond are intentional extensions — AIX assists for classification, native Dayforce payroll handoff, real-time co-employment risk — that none of the three reference platforms own end-to-end.

Architectural discipline

Per Architecture Principle 01, Contractors is a filter on every Flex Work surface, not a sibling product. Every remaining task in this list maps to an existing page or model — section 03 names the host. Where we have to add a field, we extend the contractor record (CONTRACTOR_WORKERS_RAW), the agreement store (CONTRACTOR_AGREEMENTS), the document store (CONTRACTOR_DOCS), or the invoice store (CONTRACTOR_INVOICES). Net new top-level files: zero.

04 · Admin · configure the program

The IC program is configured before a single contractor is invited.

Admin owns the catalog. The IC program card in Settings → Configuration (hosted by pages/contractor-config.jsx) puts the country contract templates, the per-country payment-method matrix, the insurance / COI policy library, the IC Compliance Officer approval router, and the background-check / COI verify / 1099 e-file / mass-payment / e-Sig connector tier under one shell. Defaults are seeded per-tenant the first time a customer opens the card; every catalog supports inline toggle / edit / restore-defaults. The Manager intake and Contractor portal both read off the same configuration, so a turn on the Admin card flows downstream within one render.

06 · Contractor · self-serve

The contractor logs in to Flex Work and runs their own onboarding, agreements, invoices, and year-end — without an Admin in the middle.

This was the biggest gap in the original v1.0 and the one Deel built their entire wedge around. Flex Work had the IC data model and the Admin / Manager surfaces but the contractor side of every flow was simulated by toasts. The fix was not a new app — it was a pool-conditional branch in pages/worker-mobile.jsx that hands off to pages/worker-mobile-ic.jsx, which renders a four-tab portal (Engagements · Invoices · Documents · Profile) inside the same iOS device frame. Each contractor-side write hits the same per-window stores Admin / Manager read from, so the rest of the surface picks up changes immediately. All eight tasks land in one file.
