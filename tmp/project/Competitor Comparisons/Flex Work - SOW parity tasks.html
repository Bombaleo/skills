Flex Work VMS · SOW parity tasks · v1.0

What it takes to ship SOW at Fieldglass and Vndly parity.

SOW is the highest-formality engagement type in Flex Work and the one furthest from the frontline shift workflow the rest of the product was built around. The data layer, dashboard lane, unified detail variant, and sample seed are in place. The four role workbenches — Admin configures the program, Manager runs the SOW, Agency delivers the work, Worker does the work — are the remaining gap. This document inventories those tasks by role, with priority, current status, parity tag, and effort, plus a three-phase sequencing recommendation at the end.

01 · Scope & methodology

SOW means MSA-anchored, deliverable-led, milestone-billed.

For the purposes of this list, a Flex Work SOW is the engagement type defined in Architecture §05 — SupplierContract subtype "SOW", anchored to an MSA, with deliverables, milestone acceptance, a fee schedule, and a change-order workflow. The benchmark is functional parity, not visual mimicry — we match what Fieldglass and Vndly can do, in the Everest visual language and the Flex Work data model.

What counts as parity

A task is parity-bound if a Fieldglass or Vndly customer can run the equivalent workflow today and we cannot. Tasks tagged Beyond are intentional extensions — AI-assist, native Dayforce integration, MSP cockpit — and ship to differentiate, not to keep up.

Architectural discipline

Per Architecture Principle 01, SOW is a filter on every Flex Work surface, not a sibling product. Every remaining task in this list maps to an existing page or model — section 03 names the host file and the existing primitives each task reuses. 03 · Architecture · embed, don't add

Every remaining task lives in an existing page or model.

SOW is a filter on every Flex Work surface, not a sibling product. Per Architecture Principle 01, the goal is to extend the surfaces we already have — Settings, Requisitions, the unified detail page, Workforce, Timesheets, Invoices, Worker mobile — rather than fork a parallel SOW app. The mapping below names the existing host file for every open task and the existing primitives it reuses. New files are flagged so we can see, at a glance, where we are actually adding architecture and where we are just adding behavior.

Rule of thumb. If a task would create more than one new top-level file, push back. Most remaining work hangs off four existing surfaces: pages/req-variants.jsx (the unified detail's SOW body — extended with interactive milestone, deliverable, change-order, and proposal accordions), pages/new-requisition.jsx (SOW-aware intake), pages/worker-mobile.jsx (assignment cards, T&M time, deliverable uploads), and pages/sow-config.jsx (Admin program card we just shipped). The data layer is already in pages/sow.jsx and stays there.

04 · Admin · configure the program

The SOW program is configured before a single requisition opens.

Admin owns the catalog. Without a template library, a fee-schedule library, an approval router, and an e-signature integration, the Manager has nothing to start an SOW from. Fieldglass ships these as a configuration tier that is genuinely hard to learn but exhaustive; Vndly ships a leaner version with sensible defaults. Flex Work needs both energies — the depth, exposed through Everest's calmer Settings surface.

The Manager is the buyer-side owner from intake to closeout.

A Manager opens an SOW, sources it to a supplier panel, evaluates proposals, signs, then spends six to twelve months tracking milestones, accepting deliverables, approving invoices, and absorbing change orders. The Manager surface decides whether SOW feels like a separate product or another lane of the same Flex Work — and we need the latter.

09 · Sequencing & phased rollout

Three phases. End-to-end parity in roughly six months.

The dependency chain is real: Manager intake can't ship before Admin templates and approval routes; Agency proposal can't ship before Manager RFx; Worker time entry can't ship before Agency time approval. The phasing below threads the chain so each phase produces a usable, demoable slice of the full SOW workflow rather than disconnected fragments.

What we are deliberately not doing

We are not building a standalone "Flex Work SOW" product. Per Architecture Principle 01 the SOW engagement type is a filter on the same surfaces every other engagement type uses — the dashboard, requisitions, workforce, timesheets, invoices, analytics. Every task in this list adds to those surfaces or extends Settings; none of them introduce a parallel SOW app, an SOW-only nav, or an SOW-only audit log. That is the design discipline that lets us reach Fieldglass and Vndly parity in thirty weeks rather than two years.
