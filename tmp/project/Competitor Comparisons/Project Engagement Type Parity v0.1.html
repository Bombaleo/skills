What Project is, in our model. Supplier-led delivery against a defined budget. The supplier names the team, the buyer reviews progress against burn, billing is T&M or fixed against a not-to-exceed cap, time is captured weekly per worker per task. No recurring schedule, no clock-punch. Distinct from SOW (MSA-anchored, milestone-billed, deliverable-led) and from Assignment (named worker, single role, period timesheet).

What we have today. After the v0.86 build: everything in Foundations and Admin ships, plus the Manager burn dashboard / roster / amendments / timesheets / documents / comments / audit / closeout, the Agency team-proposal + burn-view + roster surfaces, and the Worker mobile project landing + task-coded time entry + scope acknowledgement. The new code lives in pages/project-engagement.jsx (primitives + storage + variant registration), pages/project-engagement-views.jsx (every Project-flavoured React component the rest of the app consumes), and the host pages call into them through window.ProjectAdminPanel, window.ProjectIntakeFields, window.ProjectVariantBody, window.AgencyProjectsTab, and window.WorkerProjectScreen. Every surface gates on the engProject feature flag — with it off the components return null and the prototype is byte-identical to the pre-Project ship.

What this list is. Every remaining job to take Project from "demo shape" to "Helios can run forty live projects without leaving the product." Tasks are grouped by the four roles that touch a Project: Admin (config), Manager (buyer), Agency (supplier), and Worker. Foundations that cross roles sit in 00 · Foundations.
