How an agency staffs each engagement type — and what's left to reach parity.

A review of the one act every engagement type shares — an agency placing a named worker against a distributed requisition — graded against the competitor that owns each role. Frontline against JoinedUp, professional against Fieldglass and Vndly. Every task is anchored to an existing page, model, or data object: the goal is parity with as little new architecture as the gap allows.

What "assignment" means here. The moment an agency turns an open line into committed labor — putting a named person against a role on a requisition or engagement. It is reviewed per engagement type because the mechanic is genuinely different at each: a Shift is allocated in seconds from a roster; an Assignment is a candidate submitted, reviewed, then confirmed on a work order; a Project is a whole team proposed against a budget envelope; an SOW is a named delivery team attached to an awarded contract.

What we have today. One shared side panel — pages/assign-worker-panel.jsx — does the heavy lifting, but it is shift-shaped: pick from a workers table, capture a rate-band exception, and either Assign (agency view) or Request (everyone else). Project already has its own purpose-built flow in pages/project-engagement-views.jsx (propose team → review → swap). SOW leans on the shared panel plus an SOW Resources pool in Workforce. Assignment — the professional contingent worker — currently borrows the shift panel wholesale and is the largest single gap.

The constraint. Minimal new architecture. The spine already exists: the engagement-type matrix in engagement-type.jsx, the ContingentEngagement / WorkAssignment / JobRequisition.flexWorkOrderId objects in Architecture v1.6, the interview submittal model, the credentialing and compliance surfaces, and the project proposal components. Most tasks below are extend, not build — each names its reuse anchor.

Section 00

Shared spine · the one panel that serves every type

6 shipped

The assignment infrastructure all four types share. The single highest-leverage move in this whole doc is making the shared panel type-aware — branch its mechanic off the engagement-type matrix instead of hard-coding the shift shape. Everything downstream gets cheaper once it does.

Today

Every assignment routes through openAssignWorkerPanel(cfg) on the Interactions bus. The panel reads flexViewAsRole to choose Assign vs Request, resolves the supplier's rate-card band for the role, and writes back through cfg.onAssign / cfg.onRequest. It does not yet read cfg.engagementType — so an Assignment, a Project line, and a Shift all get the identical shift-shaped picker.

Section 02 · Professional · Fieldglass + Vndly

Assignment · candidate → work order → activation

6 shipped 1 not chasing

The widest gap. Fieldglass does not "assign" a professional worker — it runs a workflow: the supplier submits a candidate (job seeker) against a posting, the buyer reviews and may interview, a hire creates a work order, the supplier accepts it, onboarding activity items clear, then the buyer activates. Borrowing the instant shift picker collapses four governance steps into one. The good news: every step has an existing object to lean on.

Today

An Assignment-type requisition routes through the same openAssignWorkerPanel as a Shift. The agency picks a worker and it's instantly "assigned" — no candidate-submission state, no buyer review gate, no work-order acceptance, no onboarding hold. The engagement-type matrix already classifies these rows correctly (Assignment → period timesheet, no clock punch); the assignment mechanic just hasn't followed.

Shipped

Submit candidate, don't instant-assign

On an Assignment row the agency submits a worker as a candidate into a review state, not a filled slot. The interview workflow already models exactly this — ivGetSubmittals() holds candidate submittals with status. Reuse that submittal object as the assignment's pre-hire state instead of inventing one.

Section 04 · Professional · Fieldglass + Vndly

SOW · name the delivery team on an awarded contract

6 shipped

SOW is the highest-formality type: assignment is never the headline — the contract is. The supplier attaches a named delivery team (Fieldglass "Resources") to an awarded SOW; workers bill against milestones, not slots. The Resources pool and the SOW detail variant exist; the staffing action that connects them is the remaining seam.

Today

An awarded SOW carries an SOW Resources pool in Workforce, and the unified SOW detail variant shows the named team with role + allocation in its Resources accordion. The engagement-type alias already resolves these workers to Statement of Work (not Shift). What's thin is the supplier-side action of proposing the named team onto an awarded SOW and the onboarding gate around it.

Section 05

Sequencing · one spine change unlocks the rest

The order is deliberate: the type-aware panel in Phase 1 is the dependency for almost everything in Phases 2–4. Build it once, then each engagement type inherits the right mechanic.

Phase 1 · Spine

Make the panel type-aware

  • Branch the shared panel on cfg.engagementType
  • Wire the supplier-type compat filter (v0.77 Phase 4)
  • Commit writes ContingentEngagement
Phase 2 · Assignment

Close the professional gap

  • Submit-as-candidate via the submittal model
  • Work-order acceptance on flexWorkOrderId
  • Onboarding activity-items gate
Phase 3 · Shift

Scale + guardrails

  • Bulk / mass allocation
  • Availability + double-booking guard
  • Credential pre-check at assign
Phase 4 · Project + SOW

Fold in shared checks

  • SOW propose / add named resource
  • Per-candidate compat + credential
  • Allocation vs budget envelope
