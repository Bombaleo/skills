What we found. We ship Pricing Configurations as eight named groups — regular pay, pay premiums, holiday pay, employer contributions, tenure deduction, markup, skill premiums, taxes — each authored in pages/pricing-config.jsx with a free-text "Calculation" field ("13.8% · gross pay above £9,100/yr"). The supplier-contract rate card in pages/supplier-contract-sections.jsx computes billRate with the formula pay × (1 + (burden + skill) / 100) × (1 + markup / 100), where burden reads a hard-coded value off the CLASSIFICATIONS array (W-2 22%, 1099 3%, C2C 10%) and markup reads a position-or-district number off the contract.

Those two surfaces are not connected. A customer can author a pricing config that says "Employer NI 13.8%" and "Pension 3%", bind it to a supplier contract, and the engine will still compute burden as a single 22% number from a hard-coded array. The holiday-pay group, the night/weekend pay premiums, the tenure deduction, and the VAT group are documentary — they read well in the editor and they are invisible to the calc.

What this list proposes. Wire the calc engine through the pricing-config authoring surface so what an admin writes is what the engine runs, then build the visibility, comparison, and approval surfaces around that single source of truth. Twenty-three recommendations, grouped by role: Foundations (the calc itself), Admin (Pricing Configurations), Manager (Suppliers + Inbox), Agency (supplier-side rate card). Every recommendation lands in a file that already exists.

The finding

Pricing config and the bill-rate engine are not the same code path.

Shipped v0.79

The pricing-config editor authors eight named groups; the bill-rate engine reads two hard-coded constants. Closing this gap is the lever for almost every other recommendation in the doc — once each group becomes a typed primitive the engine stacks in stage order, holiday pay, shift premiums, tenure deduction, and locale-specific taxes all become first-class outputs of the rate card instead of footnotes in the editor.

As-is · today

2 stages, 2 hard-coded inputs 
billRate(p) =
  base × (1 + (burdenPct + skillPct)/100)
       × (1 + markupPct/100)

  base       = p.payRatePref || p.payRate
  burdenPct  = CLASSIFICATIONS[p.classification]
  skillPct   = Σ SKILL_PREMIUMS[p.skillIds]
  markupPct  = p.positionMarkup
            ?? district.markupPct
            ?? avg(operatingDistricts)
1 Loaded base Single multiplier mashes burden + skills into one rowAdditiveLoadPct number. Hard-coded
2 Markup Position override › district › operating average. Contract
— Holiday pay "1.5x base rate · public holidays" — display text only, not in the calc. Text
— Pay premiums "+15% · hours 22:00–06:00" — display text only, no per-shift bill rate. Text
— Employer contributions "13.8% NI / 3% pension" — not read; engine still uses W-2 22% from the array. Text
— Tenure deduction "−5% · workers <90 days" — surfaces only on the worker-tenure timeline. Text
— Taxes "20% VAT on final bill rate" — tax engine runs at invoice time, not at quote. Text

Effect. Two customers in the UK and US authoring different pricing configs see the same number from the engine. The breakdown popover shows accurate shape (pay · burden · skill · markup) but the burden is always one of three hard-coded values regardless of the eight groups the admin painstakingly configured.

To-be · staged components

6 stages, 1 typed source 
billRate(p, ctx) =
  reduce(stages, base, (acc, stage) =>
    stage.apply(acc, pricingConfig.rules, ctx))

  ctx     = { date, shift, country,
              workerTenureDays, currency }
  stages  = [base, premiums, contributions,
             skills, tenure, markup, taxes]

  // each rule is { kind, value, scope,
  //                effectiveFrom, effectiveTo }
1 Base pay Resolved from payRatePref / band · shift variant chosen by ctx.shift. Row + ctx
2 Pay premiums Night / weekend / holiday rules from group g2 + g3 · multiplicative on base. Pricing config
3 Employer contributions NI / pension / workers' comp from group g4 · supersedes the burdenPct constant. Pricing config
4 Skill premiums Per-cert uplift from group g6b · filtered by row's skillIds[]. Pricing config
5 Tenure deduction Band rule from group g5 · resolves against ctx.workerTenureDays. Pricing config
6 Markup Existing position / district / default chain — unchanged. Contract
7 Taxes VAT / GST / state tax from group g7 · final stage · respects reverse-charge. Pricing config

Effect. The eight-group editor and the engine become the same code path. The breakdown popover now shows every layer the admin authored. UK customers get an FG-style transparency build-up; US customers stack burden, skill, markup, state tax. CLASSIFICATIONS becomes the default template for new pricing configs rather than a parallel source of truth.

Section 00

Foundations · wire the engine to the editor

6 shipped

The structural work that turns Pricing Configurations from a documentary surface into the engine's input shape. Every other section in this doc rests on Foundations 1 & 2 landing first.

F1 Shipped L

Promote each pricing-config rule to a typed primitive

Today every rule in PCFG_DEFAULT_STRUCTURE carries free-text fields — { label: "Calculation", value: "13.8% · gross pay above £9,100/yr" }. Add a structured payload alongside the existing display text: { kind: "percentage" | "amount" | "multiplier" | "band", value, threshold?, scope }. Keep the display text for editor legibility; emit the typed payload for the engine. Backward-compatible: rules without a typed payload behave exactly as today.

Why. Without typed rules the engine cannot consume what the editor produces. This is the gate task — F2 through F6 all read the new shape.

Filepages/pricing-config.jsx · PCFG_DEFAULT_STRUCTURE
RefFG: Rate Component · VNDLY: Rate formula component
F2 Shipped L

Rewrite billRate as an ordered stage reducer

Today billRate(p) in pages/supplier-contract-sections.jsx is a two-line expression: base × (1 + load) × (1 + markup). Replace with reduce(stages, base, (acc, stage) => stage.apply(acc, pricingConfig.rules, ctx)) where stages is the ordered list [base, premiums, contributions, skills, tenure, markup, taxes] and each stage filters pricingConfig.rules for its kind. Falls back to the legacy burdenPct + skillPct path when no pricing config is bound to the contract.

Why. Matches Fieldglass Rates 2.0's Rate Component Group stacking model. Gives Vendly-style "every component visible" build-up without changing the row shape.

Filepages/supplier-contract-sections.jsx · billRate()
RefFG: Rate Component Group stacking
F3 Shipped M

Add shift & holiday variants to every rate row

Today one row = one bill rate. A "Production Associate" on a Sunday night actually invoices at base + 15% night + 10% weekend, but the rate card cell shows the regular number with no surface telling the manager what Sunday night costs. Extend the row to carry payVariants: [{ kind: "regular" | "night" | "weekend" | "holiday", payRate, billRate }] derived from the pricing config's premium rules at edit time. The cell still shows the regular rate; the popover from M1 expands to the four-row variant table.

Why. Manager intake quotes are currently low by the premium spread on any non-9-to-5 role. Closing this also lets timesheets resolve the right bill rate per line without re-running the engine at invoice time.

AreaSuppliers · contract
Fileextend positions[].payVariants in pages/supplier-contract.jsx
RefFG: Rate Schedule shifts · VNDLY: Premium pay variants
F4 Shipped M

Effective-date each pricing-config rule

Rate-card rows carry effectiveFrom / effectiveTo; pricing-config rules do not. A statutory NI bump on April 6 should change every bound contract on its own — today the only way is editing the rule in place, with no audit of the value it held before. Add the same date-pair to each rule and pass ctx.date through the stage reducer so the engine resolves the rule active on the work date, not the rule active on the read date.

Why. The CSV-import / Master-data flow on the rate-card side already round-trips effective dates. Pricing config is the only layer missing the primitive.

AreaSettings · Pricing
Fileextend rule shape in pages/pricing-config.jsx
RefFG: Effective-dated rate components
F5 Shipped S

Tag every rule with currency + country

A UK NI rule on a pricing config bound to a US-only contract should no-op. Today every rule applies regardless of contract locale; the same config bound to two countries quietly miscomputes. Add scope: { countries: [], currencies: [] } to the rule shape; the engine skips rules whose scope excludes the row's contract.

AreaSettings · Pricing
Fileextend rule shape in pages/pricing-config.jsx
RefVNDLY: Country-specific calculations
F6 Shipped M

Tenure-band rule resolves at calc time, not at intake

The "First 90 days −5%" tenure deduction in group g5 is visible only on the worker-tenure timeline today — not in the rate-card row. Once F1–F2 ship, the rule becomes a band primitive [{ lt: 90, value: -5 }, { lt: 180, value: 0 }, ...] and the engine resolves it against ctx.workerTenureDays at the time the bill-rate is computed. A worker crossing the 90-day mark mid-engagement sees the rate move — with the same snapshot guarantees that protect the engagement.

Section 01

Admin · Pricing Configurations

6 shipped

The program owner's surface. Once rules are typed and effective-dated (Foundations), the authoring surface needs the safety rails that any FG / VNDLY admin would expect: impact preview before save, history per rule, industry-templated configs, and a path to migrate contracts between configs without one-at-a-time clicks.

A1 Shipped M

Impact preview on every rule save

Editing a rule today is a blind action — the admin doesn't see how many supplier contracts, positions, or in-flight assignments are about to move. Before save, run a dry pass: "Changing Employer NI 13.8% → 14.2% affects 12 contracts · 168 positions · 41 active assignments · average bill rate +£0.27/hr". Block the save behind an acknowledgment when the affected-assignment count crosses a threshold the admin sets.

Why. Pricing changes are the highest-stakes config edit in the product. FG's Rates 2.0 ships a similar "Mass Update Preview"; Vendly ships it under "Apply changes".

AreaSettings · Pricing
Filepages/pricing-config.jsx · rule editor side panel
RefFG: Mass Update Preview
A2 Shipped M

Per-rule audit ledger

The supplier-contract Rate cards card carries a history drawer per row; the pricing-config rule that drives a third of that bill rate has no such record. Add the same audit primitive (timestamp, actor, before / after, reason) to every rule, surfaced as a History drawer at the bottom of each group. Reason is required on save when the rule is bound to one or more active contracts.

Section 03

Agency · supplier-side rate card

5 shipped

The supplier's surface. Parity shipped the agency read, counter-rate composer, and tenure-progression preview. These five tighten the agency's day-to-day — clearer masked views, statutory-floor watchdogs, prefilled market data on counters, and an explicit margin line where policy permits.

G1 Shipped M

Statutory-floor watchdog on every row

A pay rate authored at the contract's currency / country can silently drop below local minimum wage after a statutory update (federal, state, or city). Read the country's minimum-wage table from pages/locales.jsx and flag any payRateMin within 5% of the floor with a yellow chip; block save when below. Surfaces on both the buyer and agency views.

Why. Wage-and-hour exposure is the single largest legal-risk vector in temporary labor. The data is already on the locale; no surface today reads it.

Filepages/supplier-contract-sections.jsx + pages/locales.jsx

G2 Shipped S

"Why is this masked?" explainer on the agency rate card

Rate-visibility policy hides pay-rate cells as ••• when an org has the supplier in band-only or masked mode. Most support tickets are agency users not knowing why. Add a small info button next to every masked cell that opens a tooltip: "Your buyer has set rate visibility to band-only. Contracted bill rate is £48/hr. Margin lines appear on each invoice." Reads contract.payRateVisibility + the policy rationale field that already exists.
