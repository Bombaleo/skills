Flex Work VMS · Interview Workflow Tasks · v1.0

We have the gate. We do not yet have the workflow.

Flex Work ships an interview-required toggle, a per-role override, and a stage taxonomy that mirrors Dayforce Recruiting (Sourced → Screened → Interview → Offer → Hired). As of v0.86 all forty tasks below are shipped behind the interviews feature flag — admin surfaces live in Settings → Interview, manager surfaces in Requisitions → Interview pipeline, agency surfaces in Requisitions → Interview inbox (when viewing as Agency), and worker surfaces in the worker mobile portal under More → Interviews. Turn the flag on in Settings → Feature Flags to surface every entry point.

01 · Where Interview stands today

A gate that opens to an empty room.

The interview feature in Flex Work today is a single binary — required or not required — expressed in three places. There is no surface where the actual interview is set up, conducted, scored, or closed out. Candidates teleport from Submitted to Interview to Offer with no record of what happened in between.

Frontline today
Submitted → Interview → Offer → Hired No scheduling between these
Professional today
Sourced → Screened → Interview → Offer → Hired Panels & scores are hand-keyed prose

What ships 5 surfaces

  • Feature flag. interviews flag in feature-flags.jsx — off by default; flipping on adds the Interview-required section to the Distribution edit panel.
  • Tenant-wide toggle. NewReqInterviewSection exposes “Require interview before offer” plus per-role overrides keyed by the requisition's bookings array.
  • Workflow choice. Work Assignment Advanced panel offers automatic vs interview match-and-offer behavior — the same gate, expressed at the assignment layer.
  • Stage taxonomy. cand_advance API moves a submittal through Submitted → Screening → Interview → Offer → Hired; every transition records actor + free-text note. The vocabulary is correct — the surfaces are missing.
  • Professional pipeline. Pro candidates carry an interviews[] array with date, type, panel, status, score, and notes — rendered as a read-only list in the candidate detail. Closest thing today to a real interview record.

What does not ship yet 12 capabilities

  • No scheduling UI. Nowhere in the buyer surfaces can a manager actually set a date/time/method/location for an interview.
  • No interview-type catalog. Pro hard-codes “Screen / Panel · Tech / Panel · Exec” as strings in seed data. Admins cannot configure types.
  • No scorecard or feedback form. Numeric scores exist in the Pro seed data but no input surface backs them.
  • No reschedule / cancel flow. Once an interview is conceptually scheduled there is no way to move or kill it.
  • No calendar integration. No Outlook / Google Workspace tokens, no .ics emission, no busy-time read.
  • No supplier inbox. Agencies cannot see interview invites on their submittals or respond on the candidate's behalf.
  • No candidate portal entry. Workers do not see an interview invite anywhere — not in worker-mobile.jsx, not in professional-worker-detail.jsx.
  • No multi-round model. Stage is a single “Interview” cell; both incumbents model Round 1 / Round 2 / Final as a first-class graph.
  • No notification template management. Fieldglass calls this Letter Management; we have nothing equivalent.
  • No decline-reason taxonomy. Free text only; cannot report on why we lose candidates.
  • No interview SLA layer. Time-to-schedule, time-to-feedback, time-to-decision are not tracked.
  • No skip-interview audit. Per-role override flips off but does not require justification or approval.
02 · What Fieldglass does

A mature, dated, end-to-end interview module.

Fieldglass treats the interview as a first-class object on the submittal, not a transition on the stage graph. Every action against it — request, schedule, reschedule, cancel, conduct, score, advance — emits a record. The UX is dated; the model is right.

Request & schedule

Manager raises, supplier responds, candidate confirms.

Interview Request is a separate workflow step between Submittal and Offer. The buyer manager raises a request (interview type, proposed slots, panel, location, dial-in); the supplier accepts or proposes alternates; the candidate confirms through the supplier or, where enabled, a candidate-direct portal. Each leg is timestamped and audited.

First-class object Audited every leg Multiple proposed slots Dated UI
Letter Management

Notification templates as a configuration surface.

Every interview-related notification — invite, reminder, reschedule, cancel, outcome — is a template the admin edits in Admin → Letter Management → Recruiting Emails → Interview Requests. Templates carry token interpolation (candidate name, job, panel, slot, dial-in), brand assets, and per-supplier overrides. Audit shows which template fired against which submittal.

Token grammar Per-supplier overrides Audit per send No previewer for new admins
Feedback & scorecards

Configurable scorecards per job family.

Admins build scorecard templates — competency rows, rating scale, freeform sections — in Configuration → Interview Management. The card is bound to the interview type and required to advance the candidate from Interview to Offer. Multiple interviewers each submit their own card; the system rolls them up to a panel score with weights an admin sets.

Template per job family Required to advance Weighted roll-up No mobile entry
03 · What VNDLY does

A modern UX, lighter model, self-serve scheduling.

VNDLY took the Fieldglass model, kept the bones, and rewrote the surfaces — the win is in the scheduling UX and the candidate's seat at the table. Self-serve slot selection and one-click reschedule are the user-visible deltas everyone notices in a side-by-side demo.

Availability windows

Manager publishes slots, candidate picks.

Instead of a back-and-forth on proposed slots, the manager publishes their availability for the next 5 / 10 / 15 working days. The supplier forwards a single link; the candidate picks a slot from a self-serve calendar; the slot is held against the manager's calendar in real time. This is the feature that broke the “four emails per interview” pattern Fieldglass tolerated.

Self-serve slot pick Calendar conflict aware One-click reschedule No multi-panel-member coordination
Interview types & templates

Admin-curated types, default panels, default duration.

Admins curate a catalog — Phone Screen / Video / Technical / Panel / On-site — each carrying a default duration, default interviewer pool, default scorecard, default location pattern (Zoom auto-generated, office address from the location record, etc). Managers pick the type and the rest pre-fills.

Catalog as config Defaults reduce setup Per-type scorecards Catalog hard to extend mid-program
Candidate surface

A real portal entry the candidate uses.

Candidates land on a branded confirmation page with prep notes, panel bios, location / dial-in, calendar download, and a reschedule link. They post-interview submit an NPS-style two-question survey. This is the feature that lets VNDLY sell “candidate experience” as a line item in MSP RFPs.

Branded confirmation Auto .ics Post-interview survey No worker-mobile parity yet

Read

Three of these eighteen are cheap on Flex Work's existing data model: the interview-type catalog (Settings tab + per-org seed; same shape as Custom Fields), the decline-reason taxonomy (a code list under Settings → Policies, consumed by cand_advance), and the skip-interview audit (the toggle that exists today plus a required reason + actor signature on the audit log). The other fifteen are surface work — not data-model work — but each takes one to three weeks of design + build.
