Where the math actually runs. The bill-rate calculation lives in two places, both inside pages/supplier-contract-sections.jsx: billRate() for the resolved cell on each row, and billRateAtBand() for the min / max variants in the breakdown popover. Both are tiny pure functions that read the position row's pay rate, classification, skill IDs, and markup — and apply the same fixed three-step chain.

Where the named configuration sits. Every supplier contract carries a pricingConfig field set by the wizard's Step 3. The catalog of configurations — Manufacturing pilot, Warehouse standard, Distribution premium, Hospitality — Tier 1 — lives in pages/pricing-config.jsx as PCFG_DEFAULT_STRUCTURE: seven groups (regular pay, premiums, holiday pay, employer contributions, tenure deduction, markup, skill premiums, taxes), each rule a triple of {label, value} strings like {"Calculation": "+15% · hours 22:00–06:00"}.

What's missing. The math in the card never reads the structure. Burden is hard-coded from a small CLASSIFICATIONS lookup; skill premiums come from a static catalog; markup is the position-row override or a district average. Picking Manufacturing pilot versus Hospitality — Tier 1 on a contract changes one string field and nothing else. The seven groups are display, not data.

  pay.pref × (1 + burden% + skill%)
  × (1 + markup%)
)
burden ← CLASSIFICATIONS, skill% ← SKILL_PREMIUMS, markup ← positionMarkup or operating-district average. None of these read contract.pricingConfig.
Diagnosis · the gap in one paragraph

Pricing Configurations are catalog cards, not a computable engine.

Two parallel pricing models in one product. The seven groups in PCFG_DEFAULT_STRUCTURE describe the components of a bill rate — pay premiums, holiday pay, employer contributions, tenure deduction, markup, skill premiums, taxes. But each rule is a display triple, not a parameterised shape. When a manager picks Hospitality — Tier 1 on a contract, only the configuration's name binds; the burden, premiums, and skill uplifts that drive the bill rate are authored a second time on each position row in the editor panel.

Result. Changing the night-premium percentage in the Pricing Configuration has zero effect on any bound contract's bill rate. Two sources of truth, no wire between them. Fieldglass and Vendly have one wire; the work below closes ours in three moves: parameterise the seven groups so each rule is a typed shape, bind the configuration to the calc so changes propagate, visualise the full chain end-to-end so every role sees what the engine ran.

SAP Fieldglass · Rates 2.0Rate Components stacked in Groups.

The engine is built around three primitives that compose together:

  • Rate Components — the atomic, parameterised pieces of a bill rate (markup, fee, tax, NI, VAT). A rate component is a calculation, such as a markup or fee, used by buyers to build a buyer-specific rate engine; companies can build their own to determine bill rates.
  • Rate Component Groups — ordered stacks so the buyer controls precedence; SAP describes them as "essentially a way to stack rate components" so the engine knows whether a markup applies to base pay or to a running subtotal.
  • Custom Lookups — lookup tables that derive component values from key combinations (location, role, seniority). A rate lookup is a custom lookup with keys for Currency, Rate Category/UOM, and Site Code, used in rate component formulas to pull values from a managed table.

Rate Schedules then add timesheet rules so OT and holiday allocation are computed automatically from entered hours.

Workday VNDLYComposable Formulas + threshold enforcement.

Built around full-chain visibility and effective-dated changes:

  • Formulas — composable, reusable rules that combine into broader rate requirements; a Formula can reference other Formulas.
  • Tenure & rate management — the platform tracks tenure milestones on each worker and steps the rate at each gate. Tenure and rate management surfaces automated rate progressions and rate increases by tenure, with rate-card management updates over time.
  • Rate thresholds — spend caps enforce limits before invoiced amounts are committed.
  • Tax-rate scheduling — tax tables refresh on a cadence with audit-tracked future-effective and end-dated changes.

Vendly's stance is that the agency markup, NI/pension, holiday pay, and tax must be visible on the bill rate end-to-end — "transparency over composition" rather than "single number".

Section 01

Admin · author and apply the engine

The configuration owner is the only role authorised to bind a pricing configuration to a contract or to change one. Admins parameterise the seven groups, define the rate-component catalog, set stacking order, schedule effective dates, simulate a change before it ships, and set the program thresholds that gate manager actions. Every task in this section lands inside pages/pricing-config.jsx or extends pages/supplier-contract.jsx — the surfaces already exist; the data shape and the calc wire are what need to change. Manager and Agency surfaces in sections 02 and 03 read the engine's output; they do not modify what it runs.

P1

Rate-type per row — Pay-to-Bill, Bill-to-Pay, Factor, Coefficient

Fieldglass ships fifteen Rate Types. Four show up on every multi-country program: Pay-to-Bill (today's default), Bill-to-Pay (worker pay derived from a static bill rate — UK/EU regulated industries), Factor (a multiplier — OT, holiday, on-call — that composes with any base rate), Coefficient (stored multiplier: 1.50 = 50% markup). Per-row field on positions[]; computeBillRate branches on it. Holiday / OT / weekend rows stop being authored as duplicates — they become Factor rows pointing at the regular row.
