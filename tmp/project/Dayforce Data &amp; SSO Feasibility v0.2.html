Flex Work 2.0 · Integration feasibility · re-graded

What Dayforce can — and can't — hand to Flex Work

A read of two Dayforce integration surfaces — the legacy SOAP Web Services and the modern RESTful Web Services API — against the two goals for Flex Work 2.0: pull as much tenant data as possible, and SSO the user in a single session. v0.2 re-grades v0.1 after the REST guide. The headline: REST closes most of the data gaps — writes, schedules, time and labor are all first-class now — while the SSO answer is unchanged. Verdict up front, then a field-level inventory, the create/view/edit matrix, the limits that survive, and the pattern that works.

Goal 1 — Pull (and push) tenant data

Org & workforce data: yes — and now writable

Supported · read + write

The REST API exposes typed resources — Employee, OrgUnit, Department, Job, Position, schedules, punches, pay — each with GET to read and POST / PATCH to write. That removes v0.1's two biggest caveats: writes are no longer narrow, and schedules / time / labor are first-class endpoints, not report-only. Pagination and a rate limiter replace the 1,000-record governor.

Goal 2 — Single session for payroll admins

Admin SSO: still not through any API

Via embed handoff

REST adds token-based auth — but connect/token still takes a service-account username + password and returns a Bearer JWT for that account, not a human sign-in. No SAML / OIDC / token-acceptance for an end user exists on either surface. The single-session path stays the planned embedded iframe + signed Dayforce token handshake; the API's job is to supply each admin's Role / access context. See §05.

Section 01

Two surfaces, one tenant

Dayforce exposes the same tenant through two integration surfaces. The legacy SOAP surface is five methods on IDayforceService plus a push receiver — read / sync, employee-centric. The modern REST surface at /api/{clientName}/v1 is verb-based CRUD over dozens of typed resources, described by an OpenAPI / Swagger spec, with OAuth-style token auth. For Flex Work, REST is the primary surface; SOAP stays useful for its push-notification receiver and as a fallback. REST · /api/{client}/v1SOAP · IDayforceService

GETread

Retrieve a resource by XRefCode, or a paginated collection when no code is given. JSON responses.

POSTcreate

Add a new entity. The body mirrors the GET shape; its XRefCode must be unique or validation fails.

PATCHupdate

Change an existing resource by XRefCode. Missing field = unchanged; null field = cleared.

DELETEremove

Supported on a narrow set — e.g. employee punches and legacy labor metrics. Not a general delete.

connect/tokenauth

POST service-account creds to the identity server → a Bearer JWT, valid ~3600s. Used on every later call.

ClientMetadatadiscover

Resolves the version-specific ServiceUri so you can bypass the per-request redirect. Call ≥ once a day.

Auth is token-based and service-to-service. POST Grant_type=password with CompanyId, Username, Password and Client_Id = Dayforce.HCMAnywhere.Client to dfid.dayforcehcm.com/connect/token; you get back a JWT access_token (token_type: Bearer, expires_in: 3600). Thereafter every call carries the Bearer header — credentials aren't sent each time. HTTP Basic auth is also accepted but token auth is recommended. REST · connect/token

The token can't be revoked. The guide is explicit: access tokens are non-revocable and must be handled responsibly. Treat the service credential and any cached token as managed secrets, and lean on the ~1-hour expiry rather than expecting to invalidate one early. REST · token handling

Scope follows the account. When a request runs under a token, "the security access associated with your user account is enforced" — the same envelope as SOAP. REST adds two filters on top: Access Authorizations and Field-Level Filtering, so the integration sees exactly what its role is granted, no more. REST · Data Authorization & Filtering

The SOAP push path still exists. If event-driven sync matters, Dayforce can call a receiver you host (Receive) and you can pull / acknowledge queued events. REST has no equivalent receiver, so this stays a SOAP-side capability. SOAP · Receive

REST request lifecycle
1

GET ClientMetadata

Resolve ServiceVersion + ServiceUri for this client; cache it (refresh daily).

2

POST connect/token

Service account → Bearer JWT, ~3600s lifetime.

3

GET / POST / PATCH (Bearer)

Read paginated collections, write by XRefCode — one resource type per call.

4

Re-token on expiry

No logout to call; just request a fresh token when the current one lapses.

Setup first: a Web Services user + role (Web Services + Read Data, plus PutWorkflowValidationAction for writes), a password policy, and access authorizations — all configured in Dayforce before call one. REST · Add the Web Services Role Feature

REST ergonomics that make a sync practical

✓

Token auth

connect/token → Bearer JWT (~3600s). No credentials on each call; re-token on expiry.

✓

Pagination

Collections page through results — the documented way past the old 1,000-record employee governor.

✓

XRefCode addressing

Every resource is fetched, created and updated by a stable XRefCode — one model for read and write.

✓

Field-level filtering

Responses can be trimmed to authorized fields, so payloads stay lean and access stays scoped.

✓

OpenAPI spec

A Swagger doc at /swagger/docs/v1 generates a typed client — fewer hand-rolled request shapes.

✓

Redirect bypass

ClientMetadata hands back the version-specific URL so high-volume runs skip the per-request redirect.

Section 04 · Limits

What REST closed, and what's still open

v0.1's gaps fell into hard absences, shape constraints and manual prerequisites. The REST guide resolves the data-side ones outright. What remains is narrow: one genuine limit no API covers (end-user SSO), the service-account scope, the VMS commercial domain that is Flex Work's by design, and the manual prerequisites a Dayforce admin must complete before any call — REST adds a few of its own.

Closed by the REST API

These were v0.1 caveats against the SOAP surface. The REST guide retires them.

Closed

Writing data back is no longer narrow

v0.1: SOAP Execute only queued jobs / clock functions — no CRUD. REST is full CRUD: POST creates and PATCH updates Employee, Org Unit, Department, Job, Position, Schedules, Punches, Pay Adjustments, Labor Metrics and more, each addressed by XRefCode.

Resolved → Flex Work can provision and update a payrolled worker directly via REST instead of import-only. Mind PATCH's null = clear vs missing = unchanged rule. REST · HTTP Verbs
Closed

Schedules, time & labor are first-class

v0.1: no GetSchedulesRequest / GetTimesheetsRequest — reachable only through a configured report. REST has typed endpoints: Employee Schedules (GET / POST / PATCH), Punches (POST / PATCH / DELETE), Availability (POST), WFM Data, Labor Metrics and Operating Hours.

Resolved → read and write schedule / time data without standing up a report first. Reports remain only for genuinely custom fields. REST · Employee Schedules
Closed

The 1,000-record governor → pagination

v0.1: SOAP employee queries were capped at 1,000 records per call. REST paginates collection responses and publishes a rate limiter instead, so a large tenant is pulled page by page within documented limits.

Resolved → page through with backoff that respects the rate limiter; the uncapped XRefCode-list-then-detail pattern still applies. REST · PaginationREST · Rate Limiter

Still open — and one boundary by design

One genuine limit no API on either surface covers, one scoping rule — and the VMS commercial domain, which is Flex Work's by design, not a Dayforce shortfall.

No

End-user SSO / token exchange

REST's connect/token looks like OAuth but only mints a Bearer JWT for a service account from its username + password. There is no SAML, OIDC or token-acceptance method that signs a Flex Work user in. Neither surface can do end-user SSO.

Workaround → for payroll admins, single session comes from the Dayforce Core embed handing Flex Work a signed token; the API supplies only their authorization context (§05). REST · connect/tokenArch · Principle 06
By design

The VMS commercial domain lives in Flex Work

Suppliers / agencies, bill rates and markups, SOW / engagement objects and submittals aren't modelled by Dayforce — and that's intentional. Flex Work is the system of record for the contingent commercial layer; Dayforce isn't expected to hold it. REST does add recruiting-adjacent hooks (job postings, candidate sourcing, application statuses, background screening, assessments, I-9) where the two systems meet.

Division of responsibility → Flex Work owns the commercial domain; Dayforce supplies org / worker / job context and the recruiting hooks. No gap to close — a boundary to honour. REST · Candidate & Job Posting feeds
Scoped

You only see what the account is authorized for

Every token request runs under the service account's security, and REST layers Access Authorizations + Field-Level Filtering on top. "As much data as possible" is really "as much as this account's role and field grants allow."

Workaround → provision the integration role with the broadest defensible access and field grants at onboarding, and document the scope so gaps read as configuration, not bugs. REST · Data Authorization & Filtering

Manual prerequisites — must be configured in Dayforce first

Not API limitations — setup the customer's Dayforce admin (or Dayforce Services) must complete by hand before Flex Work can pull data or honour SSO. They belong on the onboarding checklist; until done, the integration fails.

Manual

Web Services user, role & field access

The token credentials belong to a provisioned user whose default role carries the Web Services feature with at least Read Data — plus PutWorkflowValidationAction for any write, and Notification subfeatures if used. Each resource also needs its own grants: e.g. org units require the Organization subfeature (Org Setup > Organization) and Location field-level access (RESTful Services > Configuration). A password policy and access authorizations must also be set.

On the checklist → agree the account's role, feature flags and field grants up front; anything outside them reads as missing data. REST · Add the Web Services Role Feature
Manual

Token secret handling (no revocation)

Access tokens cannot be revoked and last ~3600s. The service credential and any cached token are managed secrets; a leaked token is valid until it expires.

On the checklist → store the credential in a secret manager, cache tokens server-side only, rotate the password on a schedule, and never expose either to the browser / iframe. REST · token handling
Manual

IP allowlisting for Web Services

Flex Work's outbound IPs must be allowed under System Admin > IP Network Restrictions > Dayforce Service. Until then calls are refused — the IpAddressRejected (1006) error on the SOAP side; the same network gate fronts REST.

On the checklist → collect Flex Work's egress ranges and have the admin add the Allow rule before go-live. Guide p.313–314SOAP · error 1006
Manual

Build & secure the signed-token embed handshake

The single-session path for admins is the iframe handoff — postMessage({type:'fw:sso', token}) from Dayforce Core into Flex Work. Token format, signing-key exchange, verification and the frame-ancestors / X-Frame-Options lockdown must be built and security-reviewed. None of it exists in any API or guide.

On the checklist → own this as an embed-contract workstream with Dayforce Core; treat the signing key as a managed secret. Arch · frame contract
Manual

SSO federation & mapped admin accounts

If the payroll admins sign in to Dayforce via SSO, that federation is a Dayforce implementation activity, and SSO / MFA are mutually exclusive. Each operator must also resolve to a Dayforce user with the right Role and a populated key (LoginId / FederationId / SSO login) or the token subject can't be matched.

On the checklist → scope the admins' federation with Dayforce Services and confirm the roster carries roles + a key; spot-check it in an early sync. Guide · SSO LoginsGuide p.298
Manual

Clean, unique cross-reference codes

XRefCodes are the join key for every REST read and write — and POST requires an XRefCode that's unique or validation fails. They're configured per object in Dayforce, and the Guide warns against special characters such as hyphens.

On the checklist → agree XRefCode conventions during data mapping, before the first backfill or write. REST · POST (unique XRefCode)Guide p.421
Section 05 · Goal 2

Single session for the payroll admins

The users in scope are the admin operators who run the payroll program — a small, privileged set who are already authenticated in Dayforce Core. "Single session" means Dayforce vouching for an admin it already trusts when Flex Work opens — not a fresh sign-in. The planned mechanism (per the Architecture) is an embedded iframe + a signed Dayforce token; the API supplies the authorization context. Neither the SOAP nor the REST surface performs the sign-in — and v0.2 confirms REST's token auth doesn't change that.

How the admin single-session composes
1

Admin is signed in to Dayforce Core

Via native login or SSO at the IdP — the existing payroll-operator session. Not a Flex Work concern.

2

They open Flex Work from the left nav

It loads in an iframe in ?embed=dayforce mode; host chrome is suppressed so the boundary is invisible.

3

Dayforce hands off a signed token

postMessage({type:'fw:sso', token}) into the iframe. Flex Work verifies the signature and trusts the admin's identity — one session, no second login.

4

Authorize from synced data

The admin's Role (e.g. FlexWorkAdmin) and access context — pulled by the API — decide what they may do.

Net: the embed token answers "who is this admin"; the API sync answers "what may they do". Neither is the REST connect/token nor the SOAP Authenticate call. Arch · Principle 06

Why REST token auth isn't the SSO mechanism. connect/token takes a service account's Username + Password and returns a Bearer JWT for that account. It never accepts a human admin's Dayforce session, a SAML assertion or a third-party bearer token — so it can't sign an admin in. It's the integration's own credential, not the user's. REST · connect/tokenArch · Principle 06

What the API contributes. Once the embed token says who the admin is, Flex Work still needs what they can touch. The Employee / HR-admin reads carry each admin's Roles, access authorizations, LoginId and FederationId — enough to resolve the token subject to a Dayforce user and apply payroll-appropriate authorization. Because admins are few and known, keeping this current is cheap. REST · GET HR AdminSOAP · Roles / UserOrgs

What Dayforce handles, manually. If those admins log into Dayforce via SSO, that federation is a Dayforce implementation activity (IdP federated with Dayforce; SSO logins provisioned per user). SSO and MFA are mutually exclusive — worth confirming for privileged operators. Guide · SSO LoginsGuide p.298

The open item. The signed-token handshake itself — token format, signing-key exchange, verification, and the frame-ancestors / X-Frame-Options lockdown — is owned by the embed contract, not any API reference. It must be built and security-reviewed; this doc only marks where the boundary sits. Arch · frame contract

Section 06 · Recommendation

The integration pattern that works

Putting the two goals together, REST-first: a service account discovers the endpoint, tokens in, backfills and stays current with paginated reads, writes back where Flex Work owns the worker, and supplies the authorization context — while the admin's single session is the embed token handshake and the VMS commercial domain stays in Flex Work. Step 0 is the manual prerequisites from §04 (role + field access, token secret, IP allowlist, embed handshake, SSO federation, mapped accounts, clean XRefCodes); the steps below assume they're done.
