Flex Work 2.0 · Integration feasibility

What Dayforce can — and can't — hand to Flex Work

A read of the Dayforce SOAP Web Services API and the Dayforce Implementation Guide against the two goals for Flex Work 2.0: pull as much tenant data as possible, and SSO the user in a single session. Verdict up front, then a field-level inventory of what's reachable, what needs manual setup in Dayforce first, and the integration pattern that works around the gaps. Every claim is sourced.

Goal 1 — Pull tenant data

Org & workforce data: largely yes

Supported

The Employee object is deep — identity, org placement, jobs, employment, pay, skills, HR. Reference data (org units, departments, jobs, pay grades) and any configured report are reachable. Delta sync and org-unit scoping are built in. The ceilings are a 1,000-record governor and an employee-centric model — not a wall.

SOAP · GetEmployeesRequestGuide p.314
Goal 2 — Single session for payroll admins

Admin SSO: achievable, but not through this API

Via embed handoff

The users in scope are the admin operators running the payroll program — a small, privileged, known set. They're already signed in to Dayforce Core, so the single-session path is the planned embedded iframe + signed Dayforce token handshake — not a SOAP call. This API's job is narrower: sync each admin's Role, UserOrgs and LoginId so the token-authenticated admin is authorized correctly. See §04.

Section 01

The shape of the API

The whole surface is five methods on IDayforceService plus a push-notification receiver. Query returns lists, Execute performs single functions, and a session ticket from Authenticate threads through every call. There is no GraphQL-style field selection — you ask for a request type and get its fixed response shape back.

Authenticateauth

Company + user + password → a session ticket used by every later call.

Queryread

Returns 0..n items. The request object's type picks the query — employees, reports, notifications.

Executeact

Single result. Queues a batch job, runs a clock function, fires an alert, or runs one report.

GetClientSiteUridiscover

Resolves which site URL and service version this company should call. Step zero.

Logoutsession

Ends the session and cleans up server resources such as open report pages.

Auth is service-to-service. Authenticate(companyName, userName, password) proves a configured Web Services user, not the human in front of Flex Work. Whatever org scope and pay-adjustment groups that account can see is the data envelope for the whole integration. SOAP · Authenticate

There is also a push path. Dayforce can call a receiver you host (IDFNotificationReceiverService.Receive) as events occur, and you can pull queued events with DFNotificationsRequest / DFNotificationDataRequest and acknowledge them. That makes event-driven sync possible instead of blind polling. SOAP · Receive

Before any of this works, two things must be set up in the tenant by hand. The Guide is explicit that Web Services give "near real-time communication… some programming effort is required," gated behind System Admin > IP Network Restrictions: Flex Work's calling IPs must be allowlisted or every call is rejected — which is exactly the IpAddressRejected (1006) error the API can return. Guide p.314SOAP · error 1006

Section 04 · Limits

What we can't get (and the way around)

The constraints fall into three kinds: hard absences (the concept isn't in the API), shape constraints (it's there but awkward), and manual prerequisites (work a Dayforce admin must do in the tenant before a single call succeeds). None is fatal for Flex Work — but each shapes where data must live and what onboarding has to cover.

API & data-model gaps

Limits of the SOAP surface itself, with the practical way around each.

No

End-user SSO / token exchange

There is no SAML, OAuth, OIDC or token-acceptance method anywhere in this surface. Authenticate only takes company + user + password for a service account. This API cannot, by itself, sign a Flex Work user in.

Workaround → for the payroll admins, single session comes from the Dayforce Core embed handing Flex Work a signed token — not from this API. The API only supplies their authorization context (§05). SOAP · AuthenticateArch · Principle 06
No

The contingent-labor / VMS domain

No requisitions, no suppliers or agencies, no candidates or submittals, no bill rates / markups, no SOW or engagement objects. The API models employees who already exist in Dayforce — not the flexible-work lifecycle that precedes them.

Workaround → Flex Work remains the system of record for the contingent domain. Dayforce supplies the org / worker / job context those records hang off of.
Governor

1,000 records per employee query

GetEmployeesRequest and GetEmployeesByXRefCodeRequest are capped at 1,000 employees per call. A large tenant can't be pulled in one shot.

SOAP · governor 1,000
Workaround → the documented pattern — GetEmployeeXRefCodesRequest (uncapped) for all keys, then fetch details in batches of ≤1,000. Orchestrate with paging + backoff.
Read-only

Writing data back is narrow

Execute is for queuing batch jobs, clock functions and alerts — not general CRUD on employees or org structure. You can't push a contingent worker back as an employee through this surface.

Workaround → treat this integration as read / sync. Worker-to-Dayforce provisioning needs Dayforce's import/HR tooling or the RESTful API (a separate, modern surface that does support POST / PATCH writes), scoped separately. SOAP · ExecuteGuide · RESTful API
Report-only

Schedules, time & labor aren't first-class

There's no GetSchedulesRequest or GetTimesheetsRequest. Those data sets are only reachable through a configured report, so availability depends on what the tenant has built.

Workaround → ship a standard report definition with the integration and pull it via OpenReportPagesRequest; fall back gracefully if it isn't present.
Flat

No org-hierarchy endpoint

OrgUnit carries name, address and a store flag but no parent pointer, and there's no standalone "org tree" query. The structure arrives one worker at a time via EmployeeOrgUnits / HomeOrganization.

Workaround → assemble the location list from the worker sync, or model the true hierarchy with an org-structure report if parent links are required.
Scoped

You only see what the service account sees

Data is bounded by the Web Services user's org access and pay-adjustment groups (UserOrgs, UserPayAdjCodeGroups). "As much data as possible" is really "as much as this account is authorized for."

Workaround → provision the integration account with the broadest defensible role during onboarding, and document the scope so gaps read as configuration, not bugs. SOAP · UserOrgsGuide · Service role

Manual prerequisites — must be configured in Dayforce first

These aren't API limitations; they're setup the customer's Dayforce admin (or Dayforce Services) must complete by hand before Flex Work can pull data or honour SSO. They belong on the onboarding checklist, and until they're done the integration simply fails. All sourced to the Implementation Guide.

Manual

IP allowlisting for Web Services

Flex Work's outbound IP addresses must be added under System Admin > IP Network Restrictions > Dayforce Service with an Allow rule. Until then every call is refused — surfacing as the API's IpAddressRejected (1006).

On the checklist → collect Flex Work's egress ranges and have the admin add the Allow rule before go-live. Guide p.313–314SOAP · error 1006
Manual

A Web Services account with the right role & scope

The credentials passed to Authenticate belong to a provisioned user whose org access and pay-adjustment groups define everything the integration can see. Restricted areas and certain configuration are reachable only with a Service role, which is set up through Dayforce and limited to Dayforce Services / Support staff.

On the checklist → agree the account's role and org scope up front; anything outside it reads as missing data. Guide · Service roleSOAP · Authenticate
Manual

Build & secure the signed-token embed handshake

The single-session path for admins is the iframe handoff — postMessage({type:'fw:sso', token}) from Dayforce Core into Flex Work. Token format, signing-key exchange, signature verification and the frame-ancestors / X-Frame-Options lockdown all have to be built and security-reviewed. None of it exists in either API or Guide.

On the checklist → own this as an embed-contract workstream with Dayforce Core; treat the signing key as a managed secret. Arch · frame contract
Manual

SSO federation for the admin operators (if used)

If the payroll admins sign in to Dayforce via SSO, standing that up is a Dayforce implementation activity — the identity provider is federated with Dayforce, not toggled on self-serve. SSO and MFA are mutually exclusive: per the Guide, MFA "can't be used together with single sign-on."

On the checklist → scope the admins' SSO/SAML federation with Dayforce Services; it's a small, known group, so coverage is tractable. Guide p.298
Manual

Admin accounts mapped: roles, logins & FederationId

To authorize a token-authenticated admin, each operator must resolve to a Dayforce user with the right Role (e.g. FlexWorkAdmin) and a stable key — LoginId / FederationId / SSO login (People > Security Settings > SSO Logins). If these are blank, the token subject can't be matched and the admin lands unauthorized.

On the checklist → confirm the payroll-admin roster carries roles and a populated key, and spot-check it in an early sync. Guide · SSO LoginsSOAP · Roles / LoginId
Manual

Report definitions built in the tenant

Schedules, time and any custom fields are only reachable through a report that already exists in the tenant. There's no typed request for them, so a missing report means missing data — not an API error to debug.

On the checklist → ship a standard report spec for the customer to import, and detect its absence gracefully. SOAP · GetReportDefinitionsRequest
Manual

Clean cross-reference codes

XRefCodes are the join keys for every batch pull and for identity mapping, and they're configured per object in Dayforce. The Guide warns against special characters such as hyphens in cross-reference codes — bad codes break batching and matching.

On the checklist → confirm XRefCode conventions during data mapping, before the first full backfill. Guide p.421SOAP · XRefCode
Section 05 · Goal 2

Single session for the payroll admins

The users in scope are the admin operators who run the payroll program — not the whole workforce. That matters: they're a small, privileged set who are already authenticated in Dayforce Core. So "single session" doesn't mean a fresh sign-in — it means Dayforce vouching for an admin it already trusts when Flex Work opens. The planned mechanism (per the Architecture) is an embedded iframe + a signed Dayforce token, with this API supplying the authorization context for that admin. The SOAP surface never performs the sign-in.

How the admin single-session composes
1

Admin is signed in to Dayforce Core

Via native login or SSO at the IdP — the admin's existing payroll-operator session. Not a Flex Work concern.

2

They open Flex Work from the left nav

It loads in an iframe in ?embed=dayforce mode; host chrome is suppressed so the boundary is invisible.

3

Dayforce hands off a signed token

postMessage({type:'fw:sso', token}) into the iframe. Flex Work verifies the signature and trusts the admin's identity — one session, no second login.

4

Authorize from synced data

The admin's Role (e.g. FlexWorkAdmin) and UserOrgs — pulled by this API — decide what they may do.

Net: the token answers "who is this admin"; the SOAP sync answers "what may they do". Neither is the SOAP Authenticate call. Arch · Principle 06

Why this API isn't the SSO mechanism. Every method takes a service-account credential or a sessionTicket minted from one. Nothing here accepts a user's Dayforce session, a SAML assertion, or a bearer token — so it can't sign an admin in, and isn't meant to. The single-session handoff lives in the Dayforce Core embed contract, not the Web Services surface. SOAP · AuthenticateArch · Principle 06

What this API contributes. Once the token says who the admin is, Flex Work still needs what they can touch. The employee sync carries each admin's Roles, UserOrgs, LoginId and FederationId — enough to resolve the token subject to a Dayforce user and apply payroll-appropriate authorization. Because admins are few and known, keeping these current is cheap. SOAP · Roles / UserOrgsSOAP · Employee.LoginId

What Dayforce handles, manually. If those admins log into Dayforce via SSO, that federation is a Dayforce implementation activity (IdP federated with Dayforce; SSO logins provisioned per user). SSO and MFA are mutually exclusive, and OData reporting doesn't support SSO users — both worth confirming for privileged operators. Guide · SSO LoginsGuide p.273, p.298

The open item. The signed-token handshake itself — token format, signing key exchange, verification, and the frame-ancestors / X-Frame-Options lockdown — is owned by the embed contract, not this SOAP reference. It must be built and security-reviewed; this doc can only mark where the boundary sits. Arch · frame contract

Section 06 · Recommendation

The integration pattern that works

Putting the two goals together: a service account does a one-time backfill, stays current with deltas and notifications, and supplies the authorization context — while the admin's single session is the embed token handshake and the contingent domain stays in Flex Work. Step 0 is the manual prerequisites from §04 (IP allowlist, service account, embed handshake, SSO federation, mapped admin accounts); the steps below assume they're done.
