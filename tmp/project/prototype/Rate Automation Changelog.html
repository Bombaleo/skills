12 changes
  • Change

    Made value source a choice on every authorable rule

    The Custom (in-engine) versus Rate card source choice is no longer limited to a hardcoded set of rules — every author-owned value now carries it, while statutory-pack values stay read-only. The default is by level: a global org-wide constant defaults to Custom, a per-dimension value to Rate card. The compiled rate-card schema includes a column only for Rate card-sourced values, so flipping a value to Custom removes its column.

    Settings → Rate Engine · v0.63 DEDominic Esposito
  • Change

    Relabelled the in-engine source to Custom and noted which version it bumps

    The source option once called "Rate Engine" is now "Custom", which reads clearly against the surface name. Each value states which version a change ties to — Custom values version with the engine, Rate card values with the card — and banded rules keep their independent per-band source choice.

    Settings → Rate Engine · v0.63 DEDominic Esposito
  • Add

    Added agency engine binding with a coverage-gated lifecycle

    An agency now binds one rate engine (pinned to a version by default, or live auto-follow) in a new Pricing Configuration Contract section; the rate card resolves through that shared engine. The lifecycle is Draft, Invited, Active, Deactivated, and Invite and Activate are hard-blocked until an engine is bound, a card exists, and every position by location in scope resolves to a rate. The active contract stamps the engine and card versions it ran against.

    Agencies · v0.62 DEDominic Esposito
  • Add

    Added a coverage panel that classifies and routes pricing gaps

    Each uncovered position by location pair is listed and classified as out-of-engine-scope (fixed by a new engine version) or no-rate-row (fixed by a new rate-card version), with the fix action on the row. Editing an agency's operating scope opens a new contract version and re-runs coverage; gaps warn non-blocking in Draft and block at invite or activate.

    Agencies · v0.62 DEDominic Esposito
  • Add

    Added a pay/bill component model with named special components

    The rate engine now separates a component's treatment from who receives it. Five flags — paid to the worker, part of the taxable base, flows into the bill, agency markup applies, output tax applies — drive the math. The rule library gained ready-made components (Tips, Mileage / reimbursement, Bonus, Per-diem / allowance, One-off payout) under friendly headings; each carries a plain-language note and lands in an Additional payments group. Tips are paid but untaxed and unbilled; mileage is billed at cost with no markup.

    Settings → Rate Engine · v0.61 DEDominic Esposito
  • Add

    Settings → Rate Engine · v0.61 DEDominic Esposito
  • Add

    Added in-group conflict resolution to the rate engine

    Every rule group now declares how overlapping rules combine — all apply (stack), highest applies, lowest applies, first match applies, or only one may apply. A selector and a plain-language consequence sit on each non-locked group, the active policy prints next to the group name, and the policy travels in the compiled rate-card schema. Order is the tiebreaker within the chosen policy.
