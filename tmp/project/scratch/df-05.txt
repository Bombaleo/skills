Localize Notifications
Dayforce provides default messages for all notifications but you can localize the text in the subject
and body of notifications.
Most messages contain tokens, or variables, (such as, {0} or {1}) that Dayforce replaces with
information when the notification is sent. For example, to change the default message to alert the
user when associates are overdue for a meal or break, you can set up a message that includes
the names of the associates: "The following associates are due for a meal: {0}", where {0}
represents the names of the associates.
Before You Begin: You must assign the Application Container > Main Application UI > Localization
feature to the user roles responsible for localizing labels and notifications. See Role Features. You
also need to display the localization keys. See Show Localization Keys.
Important: Removing or modifying tokens significantly changes the meaning of the notification.
Edit tokens with caution.
To localize notifications:
1. Go to System Admin > System Notifications.

Cross-Reference Codes
Cross-reference codes, also known as reference codes and XREF codes, are used by Dayforce in
imports and exports to integrate into other applications and systems used by your organization.
Cross-reference codes can also help you identify the different elements in the application in the
scripts and processes that export your organization's data from Dayforce.
You enter cross-reference codes in Dayforce for the elements in the application that you're
creating that will be included in imports and exports, such as your organization's jobs or
departments.
As a best practice, it’s recommended that you enter a cross-reference code that logically
represents the element that you’re creating. For example, if you’re configuring a Clerk Job in
Dayforce, you might use Clerk_Job as the cross-reference code. As long as each element has a
unique code, it can be used in exports of those elements to another system.
Elements in Dayforce don’t always need cross-reference codes to work properly, unless those
elements will be used in imports or exports, in which case, you only need to include reference
codes for those elements to be imported or exported. Nevertheless, it’s best practice to include
cross-reference codes in all entries for which you can in Dayforce in case the entries are ever
used for imports or exports.

Security Configuration
Dayforce security settings are used to determine which users can access various areas of
Dayforce, such as department managers accessing Schedules but not People, and what
information they can access, create, or update.
Users need a user name to log in to Dayforce. The user name is associated with one or more user
roles. The user role determines which features the role has access to and the authorizations that
determine what the role can do in those features.
User names are configured in System Admin > User.
User roles are configured in System Admin > Roles.
Features are configured in System Admin > Roles in the Features tab.
Authorizations are configured in System Admin > Roles in the Authorizations tab.
This following topics describe these concepts in more detail:

User Roles
User roles control which features users can access in Dayforce, and they play an important part in
the security of Dayforce. User roles are assigned to user names. Security Configuration describes
the differences between user roles and user names.
You can configure as many user roles as your organization requires in System Admin > Roles. The
more granular the access requirements, the more user roles you'll need to configure.
For example, if your organization needed to provide access to a different set of features for
corporate management, location managers, assistant managers, and employees, you'd configure
four user roles, one for each of those groups.
Note that the term ESS (employee self-service) is used throughout Dayforce documentation.
Configurations can be different, depending on the needs of your organization, but ESS generally
refers to features that individual employees can access directly. These can include individual
earning statements, annual tax forms, timesheets, and various forms they can use to update tax or
other records.
When you’re creating user roles, you can arrange them in a hierarchy of parent and child roles.

In Dayforce, you can define the password policies that you want to enforce for logging in and
using the application. To help secure Dayforce user accounts, you can use several optional
security features.
You must define at least one password policy. If necessary, you can define multiple password
policies that are assigned to different user roles. For example, you could have two password
policies, so that managers and administrators are assigned a stronger password policy than
employees.
In cases where a user has multiple user roles, and the roles are assigned different password
policies, Dayforce uses the strongest password policy assigned for that user. For example, if a
user is assigned both the Administrator and Manager user roles, but the Administrator user role
has a stronger password policy, Dayforce uses the password policy associated with the
Administrator role for that user. This means that when that user tries to change their password or
their PIN, Dayforce enforces the requirements defined in the Administrator password policy.
Note: Password policies impact every user that logs in to Dayforce and any changes to the policy
come into effect immediately; typically, a password policy is defined and configured before
Dayforce is in use by end users, but if you do need to make a change to the policy after users
have been logging on, ensure that it’s communicated to them in advance.
